Recent News

My efforts have been primarily focused at axolsoft as of late. Things are very much underway putting together projects such as Steinwise, a beer discovery service. Not only are we putting together projects at axolsoft, but I am also continuing to further my own.

Among others Latria is a light-weight scripting language that I've been working on for about a year now. The intention was to create something similar in respect to Lua, but to further simplify and strip it down, making it a universal drop in scripting substitute. It is written following the ANSI-C (C90) standard, granting Latria the ability to run across a huge variety of devices. Considering that this standard was set back 1990 it's capable of running on most devices built within the past 25 years. This offers a versatile choice of platforms to run Latria off of.

The mentioning of Lua before is significant in the respect that I am a huge Lua fan! Since first discovering it I've been enamored with it's simplicity and portability. I would be the first to promote it, however, in practice, I've found it to be a bit involved when setting it up on certain platforms. On Android the next best solution to compiling Lua through the NDK was LuaJ or JNLua. To be fair, both these are very well done. Although when working with something like the JVM you assume some baggage. JNLua is limited to Lua 5.2 & JDK 1.6, assuming those versions don't match up you'd have to make some tweaks. LuaJ is fairly nifty for it's size, but you need to get comfortable with the Java-Lua api first. Both of these are good choices, but using either leaves you with additional dependencies & apis that you'll need to hook up and get used to. I would personally want to work with something that is as similar as possible in it's hookup and usage from application to application as possible. Whether web, mobile or desktop I want to be able to use my language in the same context. Better yet if one could use this same language to say, glue things together as needed?

This project has some interesting implications, but I believe it can further improved by those around me. I intend to open source this once it's ready, and would love to see it grow under community hands. So far a size of less than 100kb makes it particularly suitable for drop in replacements with minimal overhead, but when it comes down to it I'll let the developers decide.
